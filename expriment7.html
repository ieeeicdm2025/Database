实验指南(七)：Zookeeper和Kafkak

一、Zookeeper和Kafkak

1、Zookeeper

Apache Flink 是为分布式、高性能、随时可用以及准确的流处理应用程序打造的开源流处理框架，用于对无界和有界数据流进行有状态计算。



2、Kafkak

（1）批处理（DataStream API）与流处理（DataSet API）

  • 流（Stream）：Flink 最主要的处理对象是数据流，即连续不断到达的数据。流数据可以是无界的，意味着数据没有固定的终止点，流会持续不断地生成数据。Flink会不断地处理这些数据，直到程序停止。
  • 批（Batch）：Flink也支持批处理。批处理是对有限数据集的处理，可以视作有限的流。Flink 的批处理是基于流处理的优化实现的。

（2）Flink的结构

  • Client：客户端，提交作业
  • JobManager：负责作业的调度、状态管理和故障恢复。每个作业只有一个 JobManager
  • TaskManager：负责具体任务的执行。Flink 的任务是并行的，每个 TaskManager 运行多个任务槽（task slot）

（3）操作符

Flink 的流处理基于一系列的操作符，这些操作符作用于数据流中的元素。常见的操作符包括：

  • Map：对数据流中的每个元素进行转换。
  • Filter：过滤掉不符合条件的元素。
  • Filter：过滤掉不符合条件的元素。
  • KeyBy：基于某个字段进行分组，类似于 groupBy。
  • Reduce：对数据流中的元素进行归约操作，通常用于聚合计算。
  • Window：将数据流分割成小的时间窗口进行聚合或处理。

（4）窗口

窗口是流处理中用于分组和聚合事件的机制。Flink 支持多种窗口类型：

  • 滚动窗口（Tumbling Window）：不重叠的时间段窗口，数据每到达一个新的时间段就会重新计算。
  • 滑动窗口（Sliding Window）：重叠的时间段窗口，数据在滑动窗口中会不断变化。
  • 会话窗口（Session Window）：基于事件的空闲时间来划分窗口，如果事件之间的时间间隔超过设定值，窗口会关闭。

3、使用minikube部署Zookeeper与Kafka

（1）向minikube中导入所需的镜像

minikube image load bitnami/kafka:latest
minikube image load zookeeper:latest
  • 进入minikube查看是否导入
minikube ssh
docker images


（2）使用kafka-deployment.yaml文件部署Kafka与Zookeeper

kubectl apply -f kafka-deployment.yaml
  • 查看pod状态，确认已正常运行
kubectl get pods


（3）测试Kafka与Zookeeper的功能

  • 打开三个命令行窗口，并进入wsl中（三个窗口都输入一遍，都进入wsl以备用）
wsl
  • 第一个窗口中，进入Kafka所在的pod，同样的，下面的kafka-9b4f47bd5-wdmkb需替换为自己创建的pod名称
kubectl exec -it kafka-9b4f47bd5-wdmkb -- bash
  • 创建一个新的Kafka主题——test-topic
kafka-topics.sh --create --topic test-topic --bootstrap-server localhost:9092 --partitions 1 --replication-factor 1
  • 确认主题是否创建成功
kafka-topics.sh --list --bootstrap-server localhost:9092
  • 在这个窗口使用 Kafka 作为生产者发送消息
kafka-console-producer.sh --topic test-topic --bootstrap-server localhost:9092
  • 输入待发送的消息，再回车即为发送
Hello Kafka
Hello Zookeeper
  • 不要关闭第一个窗口，同时打开第二个命令行窗口，进入Kafka所在的pod，同样的，下面的kafka-9b4f47bd5-wdmkb需替换为自己创建的pod名称
kubectl exec -it kafka-9b4f47bd5-wdmkb -- bash
  • 在这个窗口使用 Kafka 作为消费者接收消息
kafka-console-consumer.sh --topic test-topic --bootstrap-server localhost:9092 --from-beginning
  • 此时可以打开第一个窗口继续输入消息并回车发送，同时在第二个窗口中也能持续接收到相应的内容，并可以使用 ctrl+c 退出发送/接收的命令行
  • 再打开第三个命令行窗口，进入Zookeeper所在的pod，同样的，下面的zookeeper-85fc45c-sqpsp需替换为自己创建的pod名称
kubectl exec -it zookeeper-85fc45c-sqpsp -- bash
  • 连接到Zookeeper
zkCli.sh -server localhost:2181
  • 往下滑动，能看到 Welcome to Zookeeper 字样
  • 由于Kafka 会在 Zookeeper 中创建一系列路径来存储主题和分区的信息，因此我们在这里列出所有的主题。这里可以看到我们最初所创建的 test-topic 主题以及用于存储消费偏移量的内部主题__consumer_offsets
